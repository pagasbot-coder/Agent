## Сессия #1 — «Первый проход» (35–40 мин)

**Цель:** baseline — онбординг → радар → commentary → 👍/👎 → одно решение.  
**Проект:** любой текущий, где вы РП (внедрение, IT, запуск, стройка с дедлайном).

### Перед сессией (5 мин)

| # | Действие |
|---|----------|
| 1 | Откройте **https://quiet-partner.vercel.app** |
| 2 | Запишите **название + фаза** (напр. «CRM для отдела продаж, фаза UAT») |
| 3 | Мысленно: **2 домена**, которые болят (напр. «Команда» 25, «Риски» 35) |
| 4 | Откройте log-шаблон во второй вкладке |

### 0:00–0:05 — Онбординг

1. Главная → **/onboarding** (или баннер «Пройдите настройку»).
2. **Название** — реальный проект.
3. **Подход поставки** — predictive / adaptive / hybrid.
4. **8 доменов** — интуитивно; два «больных» **ниже 40**.
5. Завершить → дашборд.

**Log:** время старта, название, подход.

### 0:05–0:12 — Радар

1. Domain Radar — совпадает с ощущением «где горит»?
2. Сдвинуть **1–2 оценки** в красную зону (&lt;40).
3. Callout «критичнее всего» + pulse (если есть).

**Log:** «Радар честный?» — да/нет.

### 0:12–0:22 — Health Commentary

1. **«Обновить комментарий»** → дождаться ответа (5–15 сек).
2. Прочитать commentary + **вопросы** + disclaimer.
3. Если текст содержит `(локальный режим…)` — отметить **fallback** в log.

**Log:** 1–2 предложения о чём комментарий; упомянул ли красные домены; один вопрос — понятен / бесполезен.

### 0:22–0:28 — Навигатор

1. Панель **S1–S4** → сценарий близкий ситуации.
2. **«Спросить напарника»** — 1–2 предложения ситуации по-русски.
3. Обновить commentary — **изменился ли фокус?**

**Log:** ID сценария + изменился ли текст (да/нет).

### 0:28–0:32 — Обратная связь

1. **«Полезно»** / **«Не полезно»** — честно.
2. Если N — одна фраза почему (шаблон, не тот домен, слишком общее).

**Useful = Y**, если записали **одно действие на сегодня**.

### 0:32–0:38 — Решение (опционально)

1. *«Сегодня сделаю: _____________»* — в log.
2. (Оп.) Export JSON / «Ранний доступ» → `/waitlist`.

### 0:38–0:40 — Закрытие

Заполнить **строку #1** в [`dogfood-log-template.md`](./dogfood-log-template.md).

## Сессия #2 — «Другой контекст / live LLM» (35–40 мин)

**Цель:** проверить, работает ли co-pilot на **другом типе проекта** или **другой delivery approach**; сравнить с #1.  
**Когда:** через 1–3 дня после #1.  
**Проект:** **другой**, чем в #1 (напр. #1 = клиентский внедрение → #2 = внутренний продукт; или наоборот).

### Перед сессией (5 мин)

| # | Действие |
|---|----------|
| 1 | Перечитать строку #1 в log — что было useful / не useful |
| 2 | Выбрать **второй реальный проект** (не копия #1) |
| 3 | Решить: staging **или** `npm run dev` + `.env.local` с `DEEPSEEK_API_KEY` — если в #1 был fallback, в #2 **желательно live LLM** |
| 4 | Инкогнито / новый профиль браузера → чистый онбординг |

### 0:00–0:06 — Онбординг с другим tailoring

1. Пройти онбординг для **проекта #2**.
2. **Подход поставки** — намеренно **другой**, чем в #1 (напр. #1 predictive → #2 adaptive).
3. На шаге 3 — раскрыть **глоссарий доменов** (T-037): labels понятны без D1–D8?

**Log:** название #2, подход, отличие от #1.

### 0:06–0:14 — Радар + glossary

1. Сравнить радар с #1: **другая картина** доменов?
2. Hover/tooltip на pill радара — hint помогает?
3. Сделать **один** домен red, **один** amber — не те же, что в #1.

**Log:** какой домен red; glossary полезен? Y/N.

### 0:14–0:24 — Commentary + navigator (глубже)

1. Обновить commentary **без** навигатора — baseline для #2.
2. Выбрать **другой** сценарий S1–S4, чем в #1.
3. «Спросить напарника» с **конкретной** ситуацией (имя роли/дедлайн/конфликт — без NDA).
4. Сравнить: commentary #2 **лучше/хуже/так же**, чем #1?

**Log:** S? ID; live vs fallback; сравнение с #1 (1 фраза).

### 0:24–0:30 — Feedback + persist

1. 👍/👎 на **финальном** commentary.
2. **Reload** страницы — голос сохранился? (localStorage)
3. «Отметить действие» — RU текст в audit (DevTools, in-memory до reload).

**Log:** persist 👍 OK? Y/N.

### 0:30–0:38 — Решение + export

1. **Одно действие на сегодня** для проекта #2.
2. **«Копировать снимок»** (T-042) — JSON содержит domains + disclaimer?
3. Короткая заметка: **вернулись бы через неделю** на этот проект? Y/N.

### 0:38–0:40 — Закрытие

Строка **#2** в log. Обновить сводку: сессий 2/5, useful count.

## Сессия #3 — «Повторный визит / weekly cadence» (30–40 мин)

**Цель:** симулировать **еженедельный** ритм North Star — вернуться к **тому же проекту**, что в #1 или #2, **без** полного сброса; проверить persist и «изменило ли решение».  
**Когда:** через ~7 дней после #1 (или на следующий день, если нужно ускорить gate).  
**Проект:** **тот же**, что в useful-сессии #1 или #2.

### Перед сессией (3 мин)

| # | Действие |
|---|----------|
| 1 | **Не** очищать localStorage — имитируем «вернулся через неделю» |
| 2 | Записать: что изменилось в проекте за «неделю» (1–3 буллета) |
| 3 | Открыть staging с тем же браузером, где проходили #1/#2 |

### 0:00–0:05 — Возврат

1. `GET /` — **нет** баннера «Пройдите настройку» (persist T-028).
2. Радар — старые scores; **обновить 2–3 домена** под новую реальность.
3. Header: «Настроить проект» → можно изменить профиль без полного wipe.

**Log:** persist OK; какие домены сдвинули и почему.

### 0:05–0:15 — Commentary после изменений

1. «Обновить комментарий» с **новыми** scores.
2. Commentary **заметил** сдвиг (упомянул другой домен)?
3. Если generic — попробовать navigator + «Спросить напарника» с фразой «за неделю изменилось X».

**Log:** commentary отреагировал на сдвиг? Y/N.

### 0:15–0:22 — Сравнение с прошлой сессией

1. Открыть log #1/#2 — выполнили ли **действие**, которое записали?
2. Новое **одно действие** на эту «неделю».
3. 👍/👎 — **полезнее** ли второй заход, чем первый?

**Log:** выполнено прошлое действие? Y/N/частично; useful vs #1.

### 0:22–0:30 — Доп. surface (быстро)

| Проверка | Ожидание |
|----------|----------|
| «Ранний доступ» → `/waitlist` | 200, CTA не ломается |
| `/api/health` | `ok: true`, `deepseek_api_key_configured` (если key в Vercel) |
| Export JSON | можно отправить себе в Telegram/Notes |

### 0:30–0:38 — Gate prep

1. Заполнить строку **#3** в log.
2. Сводка: **≥3 useful?** Если да — можно идти в [`m0-go-no-go-memo.md`](./m0-go-no-go-memo.md) за **Human sign-off Go**.
3. Если &lt;3 useful — запланировать #4–#5 (другие проекты или pivot-заметки).

### 0:38–0:40 — Закрытие

PM journal (опционально): дата, #3, Y/N, готовность к M0.

## Чеклист «сессия засчитана для G2→3»

- [ ] Реальный проект (не выдуманный)
- [ ] Онбординг или persist-return осмысленны
- [ ] Радар меняли → UI отреагировал
- [ ] Commentary получен (live или fallback)
- [ ] 👍/👎 нажато
- [ ] **Одно действие на сегодня** записано
- [ ] Строка в log заполнена

**≥3 сессий с «Полезно» = Y** → evidence для M0 Go.
